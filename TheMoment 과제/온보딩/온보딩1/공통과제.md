# 각 전공들 이해하기

## Frontend

사용자가 직접 보고 상호작용하는 화면을 만드는 역할이다.

### 하는 일

- 디자이너가 만든 Figma 시안을 실제 동작하는 웹/앱 화면으로 구현
- 버튼 클릭, 폼 입력, 애니메이션 등 사용자 인터랙션 처리
- 백엔드에서 받아온 데이터를 화면에 보기 좋게 렌더링
- 반응형 웹 구현 (PC, 태블릿, 모바일 등 다양한 화면 크기에 맞게 레이아웃이 자동으로 조정되도록 구현)
- 페이지 로딩 속도 최적화 (이미지 압축, 코드 분할, 캐싱 등을 활용해 사용자가 빠르게 화면을 볼 수 있도록 개선)

### 사용 기술

| 기술 | 설명 |
|------|------|
| HTML | 웹 페이지의 구조를 만드는 언어 |
| CSS | 색상, 폰트, 레이아웃 등 시각적인 스타일을 담당 |
| JavaScript | 버튼 클릭, 데이터 처리 등 동적인 기능을 구현하는 언어 |
| React / Vue / Angular | UI를 컴포넌트 단위로 관리하는 프레임워크. 버튼, 헤더처럼 재사용 가능한 UI 조각을 한 번만 만들어두고 여러 페이지에서 가져다 쓸 수 있어 유지보수가 쉬워짐 |
| TypeScript | JavaScript에 타입을 추가해 코드 안정성과 가독성을 높인 언어 |

## Backend

사용자의 눈에는 보이지 않는 부분인 데이터 처리와 비즈니스 로직을 담당한다.

### 하는 일

- 프론트엔드의 요청을 받아서 데이터를 처리하고 응답
- 데이터베이스에 데이터 저장, 조회, 수정, 삭제 (CRUD)
- 회원가입, 로그인, 결제 같은 핵심 비즈니스 로직 구현
- 보안 처리 : 비밀번호는 절대 그대로 저장하지 않고 해시(암호화)해서 저장. 로그인 후에는 JWT 토큰을 발급해서 로그인 상태를 유지함 (인증/인가)
- API 설계 및 개발 : 프론트엔드가 데이터를 요청할 수 있는 창구(엔드포인트)를 만드는 것. 예를 들어 `/api/login`이라는 주소를 만들어두면 프론트엔드가 그 주소로 요청을 보냄

### 사용 기술

| 기술 | 설명 |
|------|------|
| Node.js / Express | JavaScript 기반 백엔드 프레임워크 |
| Java / Spring | 대규모 기업 서비스에서 많이 사용하는 백엔드 |
| Python / Django, FastAPI | 빠른 개발과 AI 연동에 강점이 있는 백엔드 |
| MySQL / PostgreSQL | 데이터를 표(테이블) 형태로 저장하는 관계형 DB. 서로 관련된 데이터를 연결해서 체계적으로 관리할 수 있음 |
| MongoDB | 데이터를 JSON 형태로 저장하는 비관계형 DB. 구조가 자유로워 빠른 개발에 유리함 |
| Redis | 자주 조회되는 데이터를 메모리에 임시 저장해서 DB 부하를 줄이고 속도를 높이는 캐싱 도구 |

## 클라우드 / DevOps

서비스가 안정적으로 운영되도록 인프라를 구축하고 관리하는 역할이다.

### 하는 일

- 서버 구축 및 관리 : 직접 서버를 사는 대신 AWS, GCP 같은 클라우드 플랫폼을 활용해 서버를 구축하고 운영
- 배포 자동화 (CI/CD) : 개발자가 GitHub에 코드를 올리면 자동으로 테스트를 실행하고, 문제가 없으면 서버에 자동 배포되는 파이프라인을 구축함. 사람이 직접 서버에 올리는 수동 작업을 없애 실수를 줄이고 배포 속도를 높임
- 서버 모니터링 : 서비스가 다운되거나 오류가 발생하면 즉시 알림을 받고 빠르게 대응
- 트래픽 급증 시 서버 자동 확장 (오토 스케일링) : 평소엔 서버 1대로 운영하다가 갑자기 사용자가 몰리면 자동으로 서버를 늘려 대응. 블랙프라이데이 같은 이벤트 때 서버가 터지지 않도록 방지
- 보안 설정, 네트워크 구성, 비용 최적화

### 사용 기술

| 기술 | 설명 |
|------|------|
| AWS / GCP / Azure | 서버, 스토리지, 네트워크 등을 제공하는 클라우드 플랫폼 |
| Docker | 애플리케이션과 실행 환경을 하나의 컨테이너로 묶어서 패키징. 환경 차이 문제를 해결해줌 |
| Kubernetes | Docker 컨테이너가 여러 개일 때 자동으로 배포, 복구, 확장을 관리해주는 도구 |
| CI/CD (GitHub Actions, Jenkins) | 코드 변경 시 자동으로 테스트하고 배포하는 자동화 파이프라인 |
| Terraform | 서버, 네트워크 등 인프라 설정을 코드로 작성해서 관리. 클릭으로 설정하던 걸 코드로 자동화함 |
| Linux | 대부분의 서버에서 사용하는 운영체제 |

## Design

사용자가 서비스를 편하고 직관적으로 사용할 수 있도록 화면을 설계하는 역할이다.

### 하는 일

- 사용자 리서치 : 어떤 기능이 필요한지, 사용자가 어떤 방식으로 서비스를 쓰는지 조사하고 분석
- 와이어프레임 제작 : 색상이나 디자인 없이 화면의 레이아웃과 구조만 잡아놓은 초안 스케치
- UI 디자인 : 실제 색상, 폰트, 아이콘 등 시각적으로 보이는 요소들을 설계
- UX 설계 : 사용자가 원하는 목표(예: 상품 구매)까지 도달하는 흐름을 설계. 불필요한 단계를 줄이고 직관적으로 사용할 수 있도록 고민하는 것이 핵심
- 프로토타입 제작 : 실제로 클릭하고 이동할 수 있는 데모 화면을 만들어 개발 전에 미리 검토
- 디자인 시스템 구축 : 버튼 색상, 폰트 크기, 간격 등을 규칙으로 정해두는 것. 팀 전체가 같은 규칙을 따르면 디자인이 일관성 있게 유지되고, 개발자도 기준을 보고 빠르게 구현할 수 있음

### 사용 도구

| 도구 | 설명 |
|------|------|
| Figma | 현재 가장 많이 사용되는 UI/UX 디자인 협업 툴 |
| Adobe XD / Illustrator / Photoshop | 그래픽 작업 및 상세 디자인에 활용 |
| Zeplin | 디자이너가 만든 시안을 개발자에게 전달할 때 색상 코드, 폰트 크기, 간격 등의 정보를 자동으로 제공하는 협업 툴 |

## 프론트엔드 ↔ 백엔드 협업 방식

1. 백엔드가 먼저 API를 설계하고, 어떤 주소로 어떤 데이터를 주고받을지 **API 명세서**를 작성한다.
2. 프론트엔드는 그 명세서를 보고 화면에서 백엔드로 **HTTP 요청**을 보내는 코드를 작성한다.
3. 백엔드는 요청을 받으면 데이터베이스에서 필요한 데이터를 조회하거나 저장한 뒤, **JSON 형태**로 응답을 돌려준다.
4. 프론트엔드는 받은 JSON 데이터를 가공해서 **화면에 렌더링**한다.
5. 개발 중에는 API가 완성되기 전에 **Mock 데이터(가짜 데이터)**를 사용해서 프론트와 백이 동시에 병렬로 개발을 진행한다.
6. 최종적으로 클라우드/DevOps가 프론트엔드와 백엔드를 **서버에 배포**해서 실제 사용자가 접근할 수 있게 만든다.

# HTTP란?

HTTP(HyperText Transfer Protocol)는 웹에서 클라이언트(브라우저)와 서버가 데이터를 주고받을 때 사용하는 통신 규약이다. 웹 페이지, 이미지, JSON 데이터 등 대부분의 웹 리소스가 HTTP를 통해 전송된다.

---

## Stateless (무상태)

HTTP는 **Stateless(무상태)** 프로토콜로, 각 요청은 서로 독립적이다. 서버는 이전 요청의 상태를 기본적으로 기억하지 않기 때문에, 로그인 정보 등은 쿠키/세션/토큰으로 따로 관리한다.

---

## Connectionless (비연결성)

HTTP는 **비연결성(Connectionless)** 방식으로 동작한다. 클라이언트가 요청을 보내고 서버가 응답을 돌려주면, 그 즉시 연결이 끊어진다. 다음 요청이 올 때는 새로운 연결을 다시 맺는다.

### 비연결성의 장점

서버가 수많은 클라이언트와 동시에 연결을 유지할 필요가 없기 때문에 서버 자원을 효율적으로 사용할 수 있다. 실제로 대부분의 클라이언트는 요청을 보내고 응답을 받은 뒤 한동안 아무것도 하지 않는 경우가 많다. 연결을 계속 유지하면 그 시간 동안 서버 자원이 낭비되므로, 비연결 방식이 훨씬 효율적이다.

### 비연결성의 단점과 해결

매 요청마다 TCP 연결을 새로 맺는 과정(3-way handshake)이 반복되어 오버헤드가 발생한다. 이를 해결하기 위해 HTTP/1.1부터 **Keep-Alive** 기능이 도입되었다. Keep-Alive는 하나의 TCP 연결을 일정 시간 동안 유지하면서 여러 요청과 응답을 처리할 수 있게 해준다.

| 구분 | 설명 |
|------|------|
| 기본 동작 | 요청 → 응답 후 연결 즉시 종료 |
| Keep-Alive | 연결을 일정 시간 유지하여 여러 요청에 재사용 |
| HTTP/2 이후 | 하나의 연결로 여러 요청을 동시에 처리 (멀티플렉싱) |

> Stateless(무상태)와 Connectionless(비연결성)는 다른 개념이다.  
> 무상태는 "서버가 이전 요청 내용을 기억하지 않는다"는 것이고,  
> 비연결성은 "응답 후 네트워크 연결 자체를 끊는다"는 것이다.

## 클라이언트-서버 구조

클라이언트(브라우저, 앱)가 요청(Request)을 보내고, 서버가 응답(Response)을 돌려주는 구조이다. 역할이 분리되어 있어서, 클라이언트와 서버를 독립적으로 개발하고 배포할 수 있다.

---

## HTTP 메서드

HTTP 메서드는 클라이언트가 서버에 어떤 작업을 요청하는지를 나타내는 방식이다. 동일한 URL이라도 메서드에 따라 서버가 수행하는 동작이 달라진다. 예를 들어 `/api/posts`라는 주소에 `GET`을 보내면 게시글 목록 조회, `POST`를 보내면 새 게시글 작성이 된다.

| 메서드 | 설명 | 예시 |
|--------|------|------|
| `GET` | 리소스 조회. 서버 상태를 변경하지 않는 읽기 전용 요청 | 게시글 목록 불러오기 |
| `POST` | 새로운 리소스 생성 또는 서버로 데이터 전송 | 회원가입, 글쓰기 |
| `PUT` | 리소스를 통째로 교체. 없으면 새로 생성 | 프로필 전체 수정 |
| `PATCH` | 리소스의 일부만 수정 | 닉네임만 변경, 좋아요 수 업데이트 |
| `DELETE` | 리소스 삭제 | 게시글 삭제 |

### PUT vs PATCH

`PUT`과 `PATCH`는 모두 수정 요청이지만 동작 방식이 다르다.

`PUT`은 리소스 전체를 새 데이터로 교체한다. 예를 들어 사용자 정보가 `{ name: "김철수", age: 20, email: "a@a.com" }`일 때 `PUT`으로 `{ name: "김영희" }`만 보내면, 나머지 필드(`age`, `email`)가 모두 사라지고 `{ name: "김영희" }`만 남게 된다.

`PATCH`는 보낸 필드만 부분적으로 수정한다. 동일한 상황에서 `PATCH`로 `{ name: "김영희" }`를 보내면 `name`만 바뀌고 나머지 필드는 그대로 유지된다.

```
PUT   /api/users/1   → 전체 교체 (보내지 않은 필드는 삭제됨)
PATCH /api/users/1   → 일부 수정 (보낸 필드만 변경, 나머지 유지)
```

## HTTP 상태 코드

### 상태 코드란?

HTTP 상태 코드는 서버가 클라이언트의 요청을 처리한 결과를 3자리 숫자로 나타낸 것이다. 클라이언트는 응답에 담긴 상태 코드를 보고 요청이 성공했는지, 실패했다면 어떤 이유인지를 파악할 수 있다.

상태 코드는 첫 번째 숫자에 따라 5가지 범주로 나뉜다.

| 범주 | 의미 | 설명 |
|------|------|------|
| `1xx` | 정보 | 요청을 받았으며 처리가 진행 중 |
| `2xx` | 성공 | 요청이 정상적으로 처리됨 |
| `3xx` | 리다이렉션 | 요청을 완료하기 위해 추가 동작이 필요함 |
| `4xx` | 클라이언트 오류 | 요청이 잘못되어 서버가 처리할 수 없음 |
| `5xx` | 서버 오류 | 요청은 올바르나 서버에서 처리 중 오류 발생 |

### 주요 상태 코드

| 코드 | 이름 | 설명 |
|------|------|------|
| `200` | OK | 요청이 정상적으로 처리됨 |
| `201` | Created | 새로운 리소스가 성공적으로 생성됨 (POST 요청 성공 시) |
| `204` | No Content | 요청은 성공했으나 반환할 데이터가 없음 (DELETE 성공 시) |
| `301` | Moved Permanently | 요청한 리소스가 영구적으로 다른 URL로 이동함 |
| `304` | Not Modified | 캐시된 데이터가 최신 상태이므로 다시 전송하지 않음 |
| `400` | Bad Request | 요청 형식이 잘못되어 서버가 이해할 수 없음 |
| `401` | Unauthorized | 인증이 필요한 요청인데 인증 정보가 없거나 유효하지 않음 |
| `403` | Forbidden | 인증은 되었으나 해당 리소스에 접근할 권한이 없음 |
| `404` | Not Found | 요청한 리소스가 서버에 존재하지 않음 |
| `500` | Internal Server Error | 서버 내부에서 오류가 발생해 요청을 처리하지 못함 |
| `502` | Bad Gateway | 서버가 게이트웨이/프록시 역할을 하다가 업스트림 서버로부터 잘못된 응답을 받음 |
| `503` | Service Unavailable | 서버가 과부하 또는 점검 중이어서 현재 요청을 처리할 수 없음 |

> `401 Unauthorized`는 "로그인 안 됨", `403 Forbidden`은 "로그인은 됐지만 권한 없음"으로 구분하면 쉽다.

## HTTP 요청/응답 구조

HTTP 메시지는 크게 **Header**와 **Body**로 나뉜다.

- **Header** : 요청/응답에 대한 메타데이터. 콘텐츠 타입, 길이, 인증 정보 등이 포함된다.
- **Body** : 실제 전송되는 데이터. JSON, HTML, 파일 등이 담긴다.
