# Docker 1주차 - 도커 기초

## 1. 도커란?

### 도커 정의

Docker는 애플리케이션과 그 실행에 필요한 모든 환경(코드, 런타임, 라이브러리, 설정 등)을 **컨테이너(Container)** 단위로 패키징하여 실행할 수 있게 해주는 **오픈소스 컨테이너 플랫폼**이다. 개발 환경과 운영 환경의 차이로 발생하는 **환경 불일치 문제**를 해결하기 위해 등장하였다.

<details>
<summary>도커를 쓰는 이유</summary>

| 이유 | 설명 |
|------|------|
| 환경 일관성 | 개발 / 테스트 / 운영 서버 모두 동일한 환경으로 실행 가능 → 환경 차이로 인한 버그 제거 |
| 빠른 배포 | 이미지만 있으면 어느 서버에서든 즉시 동일한 환경으로 실행 가능 |
| 경량성 | OS를 통째로 올리는 VM과 달리 호스트 OS의 커널을 공유하므로 가볍고 빠르게 실행됨 |

</details>

## 2. 컨테이너와 이미지 기본 개념

<details>
<summary>컨테이너란?</summary>

### 컨테이너

#### 컨테이너 개념

```text
[Google 정의] 컨테이너는 소프트웨어 서비스를 실행하는 데 필요한 특정 버전의
프로그래밍 언어 런타임 및 라이브러리와 같은 종속 항목과 애플리케이션 코드를
함께 포함하는 경량 패키지

[AWS 정의] 컨테이너는 애플리케이션 코드, 라이브러리 및 클라우드에서 실행하기
위해 필요한 기타 종속성이 포함된 소프트웨어 코드 패키지
```

#### 컨테이너가 왜 필요한가

- **경량화 및 효율성** : 컨테이너는 기존 **가상 머신(VM)보다 훨씬 가볍고 빠르며**, 더 적은 자원으로 많은 애플리케이션을 운영할 수 있어 **리소스 활용도를 향상**하고 **서버 비용을 절감**할 수 있다.
- **이식성 및 일관성** : 컨테이너는 애플리케이션과 모든 종속성을 패키징하므로 **개발, 테스트, 생산 환경 간의 차이를 제거**하여 Linux, Windows, Mac 등 **운영체제에 구애받지 않고** 사실상 **어느 환경에서나 구동 가능**하도록 한다.
- **빠른 배포 및 확장성** : **컨테이너화된 애플리케이션은 기존 방식보다 빠르게 배포**되며, 밀리초 단위로 시작하여 수초 내 확장이 가능하다.
- **DevOps 및 CI/CD 용이성** : 컨테이너는 지속적 통합(CI) 및 지속적 배포(CD) 파이프라인과 연동하여 **코드 변경 사항이 자동으로 테스트되고 운영 환경에 안전하게 배포될 수 있도록 지원**한다.
- **책임 분리** : 컨테이너화를 통해 개발자는 애플리케이션 로직과 종속성에 집중하고, IT 운영팀은 배포 및 관리에 집중하는 등 책임을 명확히 분리할 수 있다.
- **애플리케이션 격리 및 보안** : 각 컨테이너는 **독립적으로 실행**되어 문제 발생 시 영향을 격리하며, **보안 취약점 노출 범위를 감소**시킬 수 있다.

#### 클라우드에서 컨테이너 활용

클라우드의 핵심은 필요할 때 자원을 늘리고, 필요 없으면 줄이는 것이다. 컨테이너는 가볍고 빠르게 실행되는 특성 덕분에 이러한 클라우드 환경에 적합하다.

##### 마이크로서비스

하나의 애플리케이션을 기능 단위로 분리하여 각각 독립된 컨테이너로 실행하는 방식이다. 예를 들어 쇼핑몰의 경우 로그인, 상품, 결제, 배송 기능을 각각 별도의 컨테이너로 운영한다.

- 특정 기능에 장애가 발생해도 다른 기능은 정상적으로 동작한다.
- 특정 기능만 독립적으로 수정 및 배포가 가능하다.

---

##### 오토스케일링

트래픽에 따라 컨테이너 수를 자동으로 늘리고 줄이는 기능이다.

- 트래픽 증가 시 컨테이너를 자동으로 추가 생성하여 대응한다.
- 트래픽 감소 시 컨테이너 수를 줄여 불필요한 비용을 절감한다.
- VM은 생성에 몇 분이 소요되지만, 컨테이너는 몇 초면 생성 가능하여 빠른 대응이 가능하다.

---

##### CI/CD 파이프라인

코드 변경 시 빌드, 테스트, 배포가 자동으로 이어지는 흐름이다.

1. 개발자가 코드를 Git에 push
2. 이미지 자동 빌드 (`docker build`)
3. 테스트 컨테이너에서 자동 테스트 수행
4. 운영 서버에 컨테이너 자동 배포

사람이 직접 배포 작업을 수행할 필요가 없어 human error를 줄일 수 있다.

---

##### 쿠버네티스

컨테이너 수가 많아질수록 관리가 어려워지는데, 이를 자동으로 관리해주는 도구이다.

| 기능 | 설명 |
|------|------|
| 자동 복구 | 컨테이너가 죽으면 자동으로 재실행 |
| 오토스케일링 | 트래픽에 따라 컨테이너 수 자동 조절 |
| 로드밸런싱 | 트래픽을 여러 컨테이너에 균등 분산 |
| 롤링 업데이트 | 서비스 중단 없이 새 버전으로 교체 |

### VM vs 컨테이너

| 항목 | VM (Virtual Machine) | 컨테이너 (Container) |
|------|----------------------|----------------------|
| OS | 각 VM마다 독립적인 Guest OS를 가짐 | 호스트 OS의 커널을 공유 |
| 시작 속도 | 느림 (OS 부팅 과정 필요, 수십 초 ~ 수 분) | 빠름 (프로세스 실행 수준, 수 초 이내) |
| 용량 | 수 GB (OS 포함) | 수 MB ~ 수백 MB |
| 격리 수준 | 강함 (하드웨어 수준 격리) | 상대적으로 약함 (프로세스 수준 격리) |
| 성능 오버헤드 | 하이퍼바이저를 거치므로 성능 손실 있음 | 호스트와 거의 동일한 성능 |
| 사용 사례 | 완전한 OS 환경이 필요한 경우 | 애플리케이션 단위 배포 및 확장 |

> VM은 **하드웨어를 가상화**, 컨테이너는 **OS를 가상화**한다고 이해하면 쉽다.

</details>

<details>
<summary>이미지란?</summary>

### 이미지 (Image)

이미지는 컨테이너를 생성하기 위한 **읽기 전용(Read-Only) 템플릿**이다. 앱 실행에 필요한 코드, 라이브러리, 런타임, 환경 변수, 설정 파일 등 모든 것이 이미지 안에 담겨 있으며, 이 이미지를 실행하면 컨테이너가 된다.

#### 특징

- **읽기 전용** : 한 번 생성된 이미지는 수정이 불가능하며, 변경이 필요한 경우 Dockerfile을 수정한 뒤 새로 빌드해야 한다.
- **레이어 구조** : 단일 파일이 아니라 여러 레이어가 쌓인 구조로 이루어져 있으며, 변경된 레이어만 새로 내려받으면 되기 때문에 저장 공간을 효율적으로 사용할 수 있다.
- **재사용성** : 여러 이미지가 동일한 레이어를 공유할 수 있어 중복 저장을 방지한다.
- **이식성** : Docker Hub 같은 Registry에 업로드하여 다른 환경에서도 동일하게 사용할 수 있다.
- **다중 컨테이너 생성** : 하나의 이미지로 여러 개의 컨테이너를 생성할 수 있다.
- **불변성(Immutability)** : 이미지는 항상 동일한 환경을 보장하기 때문에 개발, 테스트, 운영 환경 간의 차이를 제거할 수 있다.

### 컨테이너 (Container)

- 이미지를 기반으로 **실제로 실행 중인 인스턴스**
- 이미지 레이어 위에 **쓰기 가능한(Writable) 레이어**가 추가되어 실행됨
- 각 컨테이너는 독립된 파일시스템, 네트워크, 프로세스 공간을 가짐 → **완전한 격리 환경**
- 컨테이너를 **삭제하면 쓰기 레이어의 데이터도 함께 삭제됨** → 볼륨으로 해결 (3장 참고)
- 동일한 이미지로 여러 개의 컨테이너를 동시에 실행하는 것이 가능

### 이미지 → 컨테이너 관계

> 이미지는 **클래스(Class)**, 컨테이너는 **인스턴스(Instance)**  
> 하나의 이미지로 여러 컨테이너를 생성할 수 있으며, 각 컨테이너는 독립적으로 동작한다.

```
[Docker Hub] → docker pull → [이미지] → docker run → [컨테이너 1]
                                                     → [컨테이너 2]
                                                     → [컨테이너 3]
```

### 이미지 생성 흐름

이미지는 Dockerfile → Image → Container 순서로 이어지는 흐름을 가진다. 먼저 Dockerfile에 베이스 이미지, 필요한 패키지 설치, 파일 복사, 실행 명령어 등 이미지 구성 방식을 작성한다. 작성이 완료되면 `docker build` 명령어를 실행하여 이미지를 생성한다. 생성된 이미지는 `docker run` 명령어로 실행하면 컨테이너가 되며, 이때 이미지 자체는 변경되지 않고 그대로 유지된다.

생성한 이미지는 `docker push` 명령어로 Docker Hub 같은 Registry에 업로드할 수 있으며, 다른 환경이나 팀원이 `docker pull` 명령어로 동일한 이미지를 내려받아 사용할 수 있다.

### 레이어 구조

이미지는 Dockerfile의 명령어 하나하나가 레이어로 쌓이는 구조이다. `FROM`, `RUN`, `COPY` 같은 명령어가 실행될 때마다 새로운 레이어가 추가되며, 각 레이어는 이전 레이어와의 변경 사항만 저장하기 때문에 전체 이미지를 다시 저장할 필요가 없다.

이미지를 내려받을 때도 이미 보유한 레이어는 건너뛰고 변경된 레이어만 받기 때문에 네트워크와 저장 공간을 효율적으로 사용할 수 있다. 여러 이미지가 공통 레이어를 공유하는 것도 가능하여 중복 저장이 방지된다.

레이어는 빌드 캐시로도 활용된다. 이미지를 다시 빌드할 때 변경되지 않은 레이어는 캐시를 그대로 사용하고, 변경된 레이어부터 다시 빌드하기 때문에 빌드 속도를 크게 단축할 수 있다. 이 때문에 Dockerfile을 작성할 때 변경이 잦은 명령어일수록 하단에 배치하는 것이 빌드 효율 측면에서 유리하다.

### Dockerfile 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `FROM` | 베이스 이미지 지정 (모든 Dockerfile의 시작점) |
| `RUN` | 이미지 빌드 시 실행할 명령어 (새로운 레이어 생성) |
| `COPY` | 호스트의 파일을 이미지 안으로 복사 |
| `ADD` | COPY와 유사하나 URL 다운로드 및 압축 해제 기능 포함 |
| `WORKDIR` | 이후 명령어가 실행될 작업 디렉토리 설정 |
| `ENV` | 환경 변수 설정 |
| `EXPOSE` | 컨테이너가 사용할 포트 명시 |
| `CMD` | 컨테이너 시작 시 실행할 기본 명령어 |
| `ENTRYPOINT` | 컨테이너 시작 시 항상 실행되는 명령어 (CMD와 함께 사용 가능) |

#### FROM

`FROM`은 모든 Dockerfile의 첫 번째 명령어로, 빌드할 이미지의 베이스 이미지를 지정한다. 이후의 모든 명령어는 이 베이스 이미지 위에 레이어로 쌓이게 된다.

```dockerfile
FROM ubuntu:22.04
FROM node:20-alpine
FROM python:3.11-slim
```

베이스 이미지는 Docker Hub에서 제공하는 공식 이미지를 주로 사용하며, `이미지명:태그` 형식으로 버전을 명시하는 것이 권장된다. 태그를 생략하면 자동으로 `latest`가 적용되나, 버전이 고정되지 않아 재현성이 떨어지므로 운영 환경에서는 태그를 명시하는 것이 좋다.

#### RUN

`RUN`은 이미지 빌드 시점에 실행할 명령어를 지정한다. 실행할 때마다 새로운 레이어가 생성되며, 패키지 설치나 파일 생성 등 이미지 환경을 구성하는 데 사용된다.

```dockerfile
RUN apt-get update && apt-get install -y curl
RUN pip install -r requirements.txt
```

여러 명령어를 `&&`로 연결하면 하나의 레이어로 합칠 수 있어 이미지 크기를 줄이는 데 유리하다. `RUN` 명령어를 여러 줄로 나누면 그만큼 레이어가 늘어나기 때문에, 관련 있는 명령어는 하나의 `RUN`으로 묶는 것이 일반적인 관례이다.

#### COPY

`COPY`는 호스트(빌드 컨텍스트)의 파일이나 디렉토리를 이미지 내부로 복사한다.

```dockerfile
COPY app.py /app/app.py
COPY ./src /app/src
```

`ADD`와 달리 단순 복사만 수행하기 때문에 동작이 예측 가능하고, 일반적인 파일 복사에는 `COPY` 사용이 권장된다. `ADD`는 URL에서 파일을 직접 내려받거나 `.tar` 압축 파일을 자동으로 해제하는 기능이 추가되어 있지만, 그 외의 경우에는 `COPY`를 사용하는 것이 명시적이다.

#### CMD

`CMD`는 컨테이너가 시작될 때 기본으로 실행할 명령어를 지정한다. Dockerfile 내에서 한 번만 유효하며, 여러 번 작성하면 마지막 것만 적용된다.

```dockerfile
CMD ["python", "app.py"]
CMD ["nginx", "-g", "daemon off;"]
```

`CMD`는 `docker run` 실행 시 인자를 추가하면 덮어쓸 수 있다. 반면 `ENTRYPOINT`는 항상 고정으로 실행되며, `CMD`를 `ENTRYPOINT`의 기본 인자로 함께 사용하는 패턴도 자주 쓰인다.

```dockerfile
ENTRYPOINT ["python"]
CMD ["app.py"]
# docker run <이미지명>           → python app.py 실행
# docker run <이미지명> other.py  → python other.py 실행
```

### 이미지 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `docker pull <이미지명>` | Docker Hub에서 이미지 다운로드 |
| `docker build -t <이름> .` | Dockerfile로 이미지 빌드 |
| `docker build --no-cache -t <이름> .` | 캐시 없이 이미지 빌드 |
| `docker images` | 로컬에 저장된 이미지 목록 조회 |
| `docker inspect <이미지명>` | 이미지 상세 정보 조회 |
| `docker history <이미지명>` | 이미지 레이어 히스토리 조회 |
| `docker tag <이미지명> <새이름>` | 이미지에 새로운 태그 지정 |
| `docker rmi <이미지명>` | 이미지 삭제 |
| `docker image prune` | 사용하지 않는 이미지 일괄 삭제 |
| `docker push <이미지명>` | Docker Hub에 이미지 업로드 |

### 기본 명령어

```bash
docker run hello-world
docker ps
docker ps -a
docker images
```

## 3. 도커 볼륨

### 왜 볼륨이 필요한가?

컨테이너 내부의 데이터는 컨테이너가 삭제될 때 **함께 사라지는 휘발성 데이터**이다. 예를 들어 MySQL 컨테이너를 삭제하면 그 안에 저장된 DB 데이터도 전부 손실된다. 컨테이너를 새로 띄울 때마다 데이터가 초기화되므로 **영속적인 데이터 저장 공간**이 별도로 필요하며, 이 문제를 해결하기 위한 것이 **볼륨(Volume)**이다.

### 볼륨이란?

> 컨테이너 외부(호스트 OS 또는 도커가 관리하는 영역)에 데이터를 저장하여,  
> 컨테이너가 삭제되거나 재시작되어도 데이터를 유지시켜주는 **영구 저장소**

<details>
<summary>도커 볼륨</summary>

## 도커 볼륨

### 정의

볼륨은 컨테이너의 데이터를 영구적으로 저장하기 위한 Docker의 데이터 관리 메커니즘이다. 컨테이너는 기본적으로 삭제되면 내부 데이터도 함께 소멸되는 휘발성을 가지기 때문에, DB 데이터나 로그 파일처럼 컨테이너 생명주기와 무관하게 보존되어야 하는 데이터를 저장하기 위해 볼륨을 사용한다.

---

### 특징

- **데이터 영속성** : 컨테이너가 삭제되더라도 볼륨에 저장된 데이터는 유지됨
- **컨테이너 독립성** : 볼륨은 컨테이너와 독립적으로 존재하며, 여러 컨테이너가 동일한 볼륨을 공유할 수 있음
- **Docker 관리** : Docker가 직접 볼륨의 생성과 저장 위치를 관리하기 때문에 사용자가 호스트 경로를 직접 신경 쓸 필요가 없음
- **이식성** : 볼륨은 Docker 명령어로 백업, 복원, 이전이 가능하여 운영 환경 간 데이터 이동이 용이함
- **성능** : Bind Mount 방식에 비해 Docker가 최적화된 방식으로 관리하기 때문에 I/O 성능이 더 우수함

---

### 데이터 저장 방식

Docker에서 데이터를 저장하는 방식은 세 가지로 나뉜다.

**Volume**은 Docker가 직접 관리하는 저장 방식으로, 데이터는 호스트의 `/var/lib/docker/volumes/` 경로에 저장된다. Docker가 저장 위치를 관리하기 때문에 사용자가 호스트 경로를 직접 지정할 필요가 없으며, 운영 환경에서 가장 권장되는 방식이다.

**Bind Mount**는 호스트의 특정 디렉토리를 컨테이너에 직접 마운트하는 방식이다. 사용자가 호스트 경로를 직접 지정하기 때문에 소스 코드처럼 호스트와 컨테이너 간 실시간 파일 공유가 필요한 개발 환경에서 주로 사용된다.

**tmpfs Mount**는 데이터를 디스크가 아닌 호스트의 메모리에 임시로 저장하는 방식이다. 컨테이너가 종료되면 데이터가 소멸되며, 민감한 임시 데이터를 디스크에 남기지 않아야 할 때 사용된다.

| 방식 | 저장 위치 | 데이터 유지 | 권장 환경 |
|------|----------|------------|----------|
| Volume | Docker 관리 경로 | 영구적 | 운영(Production) |
| Bind Mount | 사용자 지정 호스트 경로 | 영구적 | 개발(Development) |
| tmpfs Mount | 메모리 | 컨테이너 종료 시 소멸 | 민감한 임시 데이터 |

---

### 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `docker volume create <볼륨명>` | 볼륨 생성 |
| `docker volume ls` | 볼륨 목록 조회 |
| `docker volume inspect <볼륨명>` | 볼륨 상세 정보 조회 |
| `docker volume rm <볼륨명>` | 볼륨 삭제 |
| `docker volume prune` | 사용하지 않는 볼륨 일괄 삭제 |
| `docker run -v <볼륨명>:<컨테이너 경로>` | Named Volume 마운트 |
| `docker run -v <호스트 경로>:<컨테이너 경로>` | Bind Mount |
| `docker run --mount source=<볼륨명>,target=<컨테이너 경로>` | --mount 방식으로 마운트 |

---

## 바인드 마운트 (Bind Mount)

### 정의

바인드 마운트는 호스트 머신의 특정 디렉토리 또는 파일을 컨테이너 내부 경로에 직접 연결하는 방식이다. 볼륨과 달리 Docker가 경로를 관리하지 않으며, 사용자가 호스트의 절대 경로를 직접 지정하여 마운트한다.

```bash
docker run -v /home/user/myapp:/app node:20
docker run -v $(pwd):/workspace python:3.11
```

---

### 특징

- **실시간 동기화** : 호스트와 컨테이너가 동일한 파일을 공유하므로, 호스트에서 파일을 수정하면 컨테이너에 즉시 반영된다.
- **호스트 경로 의존** : 지정한 호스트 경로가 존재해야 하며, 경로가 다른 환경에서는 동작하지 않을 수 있다.
- **Docker 비관리** : Docker가 데이터를 직접 관리하지 않으므로 `docker volume ls`로 조회되지 않는다.
- **개발 환경 적합** : 소스 코드를 컨테이너에 실시간으로 반영해야 하는 개발 환경에서 주로 사용된다.

---

### 볼륨 vs 바인드 마운트

| 항목 | Volume | Bind Mount |
|------|--------|------------|
| 저장 위치 | Docker 관리 경로 (`/var/lib/docker/volumes/`) | 사용자가 지정한 호스트 경로 |
| 경로 지정 | Docker가 자동 관리 | 사용자가 절대 경로 직접 지정 |
| Docker 관리 | O (volume 명령어로 관리 가능) | X |
| 이식성 | 높음 (환경에 관계없이 동일하게 동작) | 낮음 (호스트 경로에 의존) |
| 성능 | 높음 (Docker 최적화) | 상대적으로 낮음 |
| 실시간 동기화 | X (컨테이너 내부에서만 변경) | O (호스트 ↔ 컨테이너 즉시 반영) |
| 권장 환경 | 운영(Production) | 개발(Development) |

> 운영 환경에서는 데이터 안정성과 이식성이 중요하므로 **볼륨**을 사용하고,  
> 개발 환경에서는 코드 변경 사항을 즉시 반영해야 하므로 **바인드 마운트**를 사용한다.

</details>

## 4. 도커 네트워크

### 컨테이너 통신 개념

> 컨테이너는 기본적으로 **격리된 네트워크 환경**에서 실행된다.  
> 도커는 컨테이너 실행 시 자동으로 **가상 네트워크 인터페이스**를 생성하여  
> 컨테이너에 IP를 부여하고, 컨테이너 간 혹은 외부와의 통신을 관리한다.

### bridge 네트워크란?

- 도커 설치 시 자동으로 생성되는 **기본 네트워크 드라이버**
- 컨테이너를 별도 네트워크 설정 없이 실행하면 자동으로 bridge 네트워크에 연결됨
- 도커는 호스트 안에 **가상 스위치(docker0)**를 만들고, 컨테이너에 가상 IP(172.17.x.x)를 부여
- 같은 bridge 네트워크 안의 컨테이너끼리는 IP를 통해 서로 통신 가능
- 외부(호스트 바깥)에서 컨테이너로 직접 접근은 불가 → **포트 포워딩으로 연결해야 함**

```
[외부 요청] → 호스트 8080포트 → (포트 포워딩) → 컨테이너 80포트
```

### 포트 포워딩 예시

```bash
docker run -p 8080:80 nginx
docker run -p 8080:80 -p 443:443 nginx
```

## 도커 네트워크 (Docker Network)

### 정의

도커 네트워크는 컨테이너 간 통신 및 외부와의 통신을 제어하는 가상 네트워크이다. 컨테이너는 기본적으로 격리된 네트워크 환경을 가지며, Docker Network를 통해 컨테이너끼리 통신하거나 외부 네트워크와 연결할 수 있다.

---

### 특징

- **격리성** : 각 컨테이너는 독립된 네트워크 환경을 가지며, 명시적으로 같은 네트워크에 연결하지 않으면 서로 통신할 수 없음
- **DNS 자동 등록** : 사용자 정의 네트워크에서는 컨테이너 이름이 자동으로 DNS에 등록되어 IP 대신 컨테이너 이름으로 통신할 수 있음
- **다중 네트워크 연결** : 하나의 컨테이너가 여러 네트워크에 동시에 연결될 수 있음
- **네트워크 드라이버** : 목적에 따라 다양한 네트워크 드라이버를 선택하여 사용할 수 있음

---

### 네트워크 드라이버 종류

| 드라이버 | 설명 | 사용 환경 |
|---------|------|----------|
| `bridge` | Docker 기본 네트워크. 가상 브리지를 생성하여 컨테이너를 연결 | 단일 호스트에서 컨테이너 간 통신 |
| `host` | 컨테이너가 호스트의 네트워크를 그대로 사용 | 포트 포워딩 없이 호스트 네트워크를 직접 사용할 때 |
| `none` | 네트워크를 완전히 비활성화 | 외부 통신이 필요 없는 컨테이너 |
| `overlay` | 여러 Docker 호스트에 걸쳐 컨테이너를 연결 | Docker Swarm, Kubernetes 환경 |

#### host 네트워크

`host` 드라이버는 컨테이너가 별도의 네트워크 네임스페이스를 갖지 않고 호스트의 네트워크 스택을 직접 공유하는 방식이다. 컨테이너가 호스트와 동일한 IP와 포트를 사용하기 때문에 포트 포워딩(`-p`) 없이도 외부에서 바로 접근할 수 있다.

```bash
docker run --network host nginx
# 호스트의 80번 포트로 nginx에 직접 접근 가능
```

포트 포워딩 과정이 없기 때문에 네트워크 성능이 가장 높다는 장점이 있다. 다만 컨테이너가 호스트 네트워크를 그대로 공유하므로 포트 충돌이 발생할 수 있으며, 네트워크 격리가 되지 않아 보안 측면에서는 주의가 필요하다. Linux에서만 완전하게 동작하며, macOS와 Windows에서는 지원이 제한된다.

#### none 네트워크

`none` 드라이버는 컨테이너의 모든 네트워크 인터페이스를 비활성화한다. 루프백(`lo`) 인터페이스만 존재하며, 외부 네트워크 및 다른 컨테이너와의 통신이 완전히 차단된다.

```bash
docker run --network none alpine
```

외부와 완전히 격리된 환경이 필요한 경우, 예를 들어 네트워크 접근 없이 파일 처리나 연산만 수행하는 배치 작업, 또는 보안상 네트워크 차단이 요구되는 작업에 사용된다.

#### overlay 네트워크

`overlay` 드라이버는 서로 다른 Docker 호스트(물리 서버)에 분산된 컨테이너들을 하나의 가상 네트워크로 연결하는 방식이다. 각 호스트의 네트워크 위에 가상의 네트워크 레이어를 덮어씌우는(overlay) 방식으로 동작하며, 컨테이너 간 통신이 호스트를 넘어 가능해진다.

```bash
# Docker Swarm 초기화 후 overlay 네트워크 생성
docker swarm init
docker network create --driver overlay my-overlay
```

Docker Swarm이나 Kubernetes처럼 여러 호스트에 걸쳐 컨테이너를 운영하는 **클러스터 환경**에서 사용된다. 단일 호스트 환경에서는 사용할 필요가 없으며, Swarm 모드가 활성화되어 있어야 한다.

---

### 기본 브리지 vs 사용자 정의 브리지

Docker를 설치하면 기본적으로 `bridge` 네트워크가 생성된다. 기본 브리지 네트워크에서는 컨테이너 이름으로 DNS 통신이 지원되지 않아 IP 주소를 직접 사용해야 한다. 반면 사용자가 직접 생성한 사용자 정의 브리지 네트워크에서는 컨테이너 이름이 자동으로 DNS에 등록되어 IP 대신 컨테이너 이름으로 통신할 수 있다. 이 때문에 운영 환경에서는 사용자 정의 네트워크 사용이 권장된다.

---

### 주요 명령어

| 명령어 | 설명 |
|--------|------|
| `docker network ls` | 네트워크 목록 조회 |
| `docker network create <네트워크명>` | 사용자 정의 네트워크 생성 |
| `docker network inspect <네트워크명>` | 네트워크 상세 정보 조회 |
| `docker network rm <네트워크명>` | 네트워크 삭제 |
| `docker network prune` | 사용하지 않는 네트워크 일괄 삭제 |
| `docker run --network <네트워크명> <이미지명>` | 특정 네트워크에 연결하여 컨테이너 실행 |
| `docker network connect <네트워크명> <컨테이너명>` | 실행 중인 컨테이너를 네트워크에 연결 |
| `docker network disconnect <네트워크명> <컨테이너명>` | 컨테이너를 네트워크에서 분리 |

</details>
