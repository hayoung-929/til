# TIL — Networking 총정리

## 1. 인터넷 / 인트라넷 / 엑스트라넷

**인터넷**
- 전 세계 컴퓨터가 TCP/IP로 연결된 공개 글로벌 네트워크
- 특정 소유자 없음, 누구나 접근 가능

**인트라넷**
- 기업·조직 내부에서만 사용하는 사설 네트워크
- 인터넷 기술(웹, 이메일)을 그대로 쓰되 외부엔 비공개
- ex) 사내 HR 시스템, 그룹웨어
- 내부 기밀 보호 + 속도·보안 향상이 목적

**엑스트라넷**
- 인트라넷을 특정 외부 파트너에게만 제한적으로 개방한 네트워크
- 인트라넷보다 넓고, 인터넷보다 좁음
- ex) 제조사가 납품업체에게만 재고 시스템 접근 허용

## 2. LAN / WAN / 이더넷

**LAN** (Local Area Network)
- 좁은 지역(건물, 사무실, 학교) 연결
- 속도 빠름 (1Gbps~10Gbps), 오류율 낮음
- 주로 이더넷·Wi-Fi 사용

**WAN** (Wide Area Network)
- 넓은 지역(도시, 국가, 대륙 간) 연결
- LAN들을 서로 연결한 개념 → 인터넷이 가장 큰 WAN
- 통신사 인프라(광케이블, 위성) 사용
- LAN = 집 안 방들 연결 / WAN = 전국 집들을 도로로 연결

**이더넷** (Ethernet, IEEE 802.3)
- 현재 LAN의 표준 유선 네트워크 기술
- 프레임 단위로 데이터 전송
- 10Mbps → 현재 10Gbps까지 발전
- 설치 쉽고 비용 저렴하며 CSMA/CD 구조가 단순·효율적이라 표준이 됨

## 3. CSMA/CD & 토큰링

**CSMA/CD** (Carrier Sense Multiple Access / Collision Detection)

여러 장치가 하나의 케이블을 공유할 때 충돌을 처리하는 방식

| 단계 | 설명 |
|---|---|
| Carrier Sense | 전송 전 케이블이 비어있는지 감지 |
| Multiple Access | 비어있으면 누구나 전송 시작 가능 |
| Collision Detection | 충돌 발생 시 JAM 신호 전송 → 전원 중단 → 랜덤 대기 후 재전송 |

- 랜덤 대기 이유 → 같은 시간 대기하면 또 충돌남. 랜덤해야 재발 확률이 줄어듦
- 이 알고리즘을 **Binary Exponential Backoff** 라고 함
- 현재 스위치 기반 환경에선 포트마다 전용 대역폭이 있어 충돌 자체가 없음 → CSMA/CD 사실상 비활성

**토큰링** (Token Ring, IEEE 802.5)
- 링(원형) 구조로 토큰을 가진 장치만 전송 가능
- 충돌 자체가 발생 불가 → 사전예방 방식
- 단점: 네트워크가 한산해도 토큰을 기다려야 해서 지연 발생
- 현재 거의 미사용

| | CSMA/CD | 토큰링 |
|---|---|---|
| 충돌 처리 | 사후 감지 후 재전송 | 사전예방 (충돌 불가) |
| 현재 사용 | O (이더넷 표준) | X (거의 미사용) |

## 4. 서브넷 / MAC / DNS

**서브넷** (Subnet)
- 하나의 큰 IP 네트워크를 여러 작은 네트워크로 분할하는 것
- 서브넷 마스크로 어디까지가 네트워크 주소인지 구분
- ex) `192.168.1.0/24` → 앞 24비트 네트워크, 뒤 8비트 호스트 → 호스트 254개 가능
- 서브넷 마스크 표현: `255.255.255.0`

쓰는 이유:
- IP 낭비 방지 → 부서별로 필요한 만큼만 할당
- 보안 강화 → 서브넷 간 트래픽을 라우터에서 필터링
- 브로드캐스트 범위 축소 → 범위가 넓을수록 네트워크 부하 증가

**MAC 주소** (Media Access Control)
- NIC에 제조 시 영구적으로 부여되는 48비트 물리적 주소
- 형식: `AA:BB:CC:DD:EE:FF`
- 앞 24비트 = 제조사 식별(OUI) / 뒤 24비트 = 고유 번호
- IP(논리적·변경 가능) vs MAC(물리적·고정)
- 같은 LAN 내 실제 전달은 MAC 기반, IP는 알지만 MAC 모를 때 → ARP로 알아냄

**DNS** (Domain Name System)
- 도메인 이름 → IP 주소 변환. 인터넷의 전화번호부
- 기본 포트: UDP 53 (응답이 크면 TCP로 전환)

동작 흐름:
1. `www.google.com` 입력 → 로컬 캐시 확인
2. 없으면 DNS 서버에 질의
3. DNS 서버가 Root → TLD(.com) → 권한 DNS 순으로 재귀 질의
4. 찾은 IP를 브라우저에 반환 → HTTP 연결 시작

- IP로 바로 안 쓰는 이유: 숫자 못 외움 + 서버 IP가 바뀌어도 도메인은 유지돼서 유연함

## 5. IPv4 / IPv6 / ARP / RARP

**IPv4**
- 32비트, 약 43억 개(2³²) 주소
- 형식: `192.168.0.1` (점으로 구분된 4개의 8비트 10진수)

| 클래스 | 범위 | 용도 |
|---|---|---|
| A | 1~126 | 대형 기관 |
| B | 128~191 | 중형 기관 |
| C | 192~223 | 소형 기관 |
| D | 224~239 | 멀티캐스트 |
| E | 240~255 | 실험용 |

- 현재 주소 고갈 → NAT(사설IP → 공인IP 변환)로 버티는 중

**IPv6**
- 128비트, 약 3.4×10³⁸개 주소 (사실상 무한)
- 형식: `2001:0db8:85a3::8a2e:0370:7334` (16진수 8그룹)

| 항목 | IPv4 | IPv6 |
|---|---|---|
| 주소 길이 | 32비트 | 128비트 |
| NAT | 필요 | 불필요 |
| 헤더 | 복잡 | 단순화 (처리 속도↑) |
| 보안 | IPSec 선택 | IPSec 기본 내장 |
| 브로드캐스트 | 있음 | 없음 (멀티캐스트로 대체) |

**ARP** (Address Resolution Protocol)
- IP → MAC 변환. 같은 LAN 내에서 IP는 알지만 MAC 모를 때 사용

동작 흐름:
1. ARP Request → "이 IP 가진 사람 MAC 알려줘" 브로드캐스트
2. ARP Reply → 해당 장치만 자신의 MAC을 유니캐스트로 응답
3. 결과를 ARP 캐시에 저장 → 다음엔 다시 안 물어봄

- ARP 스푸핑: 악의적인 장치가 거짓 ARP 응답으로 트래픽을 가로채는 공격 → MITM(중간자 공격)의 기반

**RARP** (Reverse ARP)
- MAC → IP 변환 (ARP의 반대)
- 디스크 없는 워크스테이션 부팅 시 사용하던 방식
- 현재는 DHCP가 완전히 대체

## 6. 3-way / 4-way Handshake

**3-way Handshake** — TCP 연결 수립

| 단계 | 방향 | 내용 |
|---|---|---|
| 1. SYN | 클라이언트 → 서버 | "연결하고 싶어. 내 ISN은 X" |
| 2. SYN-ACK | 서버 → 클라이언트 | "OK(ACK=X+1). 나도 연결. 내 ISN은 Y" |
| 3. ACK | 클라이언트 → 서버 | "받았어(ACK=Y+1). 연결 완료" |

- ISN 교환 이유 → 패킷 순서 추적해서 유실·순서 오류 감지하기 위해
- 왜 3번? → 2번만 하면 서버만 클라이언트를 확인한 것. 3번째 ACK로 양방향 모두 확인
- "나 너한테 보낼 수 있고, 너도 나한테 보낼 수 있다"를 서로 확인하는 최소 횟수가 3번

**4-way Handshake** — TCP 연결 종료

| 단계 | 방향 | 내용 |
|---|---|---|
| 1. FIN | 클라이언트 → 서버 | "나 더 보낼 거 없어. 끊자" |
| 2. ACK | 서버 → 클라이언트 | "알겠어. 근데 나 아직 보낼 거 있어" |
| 3. FIN | 서버 → 클라이언트 | "나도 다 보냈어. 끊을게" |
| 4. ACK | 클라이언트 → 서버 | "알겠어" → TIME_WAIT 진입 |

- 왜 4번? → 종료 시엔 서버가 보낼 데이터가 남아있을 수 있어서 ACK와 FIN을 동시에 못 보냄. 그래서 분리 → 4번
- TIME_WAIT 이유 → 마지막 ACK 유실 시 서버가 FIN 재전송할 수 있음. 2*MSL(약 60~120초) 대기

| | 3-way | 4-way |
|---|---|---|
| 목적 | 연결 수립 | 연결 종료 |
| 횟수 | 3번 | 4번 |
| 플래그 | SYN / SYN-ACK / ACK | FIN / ACK / FIN / ACK |
| 특이사항 | ISN 교환 | TIME_WAIT 존재 |
| 왜 그 횟수 | 양방향 확인 최소 횟수 | 서버 잔여 데이터 전송 보장 |

## 7. 유니캐스트 / 브로드캐스트 / 멀티캐스트 / 애니캐스트

**유니캐스트** (Unicast)
- 특정 하나의 목적지에게만 전송하는 1:1 통신
- 웹 브라우징, 이메일, 파일 다운로드 등 대부분의 인터넷 트래픽

동작 방식:
- 송신자가 목적지 IP·MAC을 지정해서 패킷 전송
- 스위치는 MAC 테이블 보고 해당 포트로만 전달 → 나머지 포트엔 전달 안 함
- 수신 측 NIC는 목적지 MAC이 자신과 일치하는 패킷만 CPU로 올림 → 불일치 시 하드웨어 레벨에서 바로 폐기

핵심 특징:
- NIC에서 하드웨어 레벨로 필터링되기 때문에 수신 대상이 아닌 PC는 CPU 부하가 전혀 없음
- 브로드캐스트는 모든 PC가 CPU까지 올려서 처리해야 하는 것과 근본적으로 다름
- 스위치 + NIC 조합이 불필요한 트래픽을 하드웨어 레벨에서 완전히 차단 → 네트워크 규모가 커져도 효율적

단점:
- 동일 데이터를 여러 명에게 보낼 때 수신자 수만큼 패킷 복사 후 전송해야 함
- 100명에게 같은 영상 스트리밍 시 동일 패킷 100번 전송 → 대역폭 100배 소모
- 이 문제를 해결하기 위해 멀티캐스트가 존재

**브로드캐스트** (Broadcast)
- 같은 네트워크 내 모든 장치에게 전송
- 목적지 IP: `255.255.255.255` 또는 `192.168.1.255`
- 목적지 MAC: `FF:FF:FF:FF:FF:FF`
- 모든 PC가 무조건 CPU까지 올려서 처리해야 함 → 과도하면 네트워크 전체 부하 (브로드캐스트 스톰)
- 라우터를 넘어가지 않음 → 라우터가 브로드캐스트 도메인을 분리
- ex) ARP Request, DHCP Discovery
- IPv6엔 브로드캐스트 없음 → 멀티캐스트로 대체

**멀티캐스트** (Multicast)
- 특정 그룹에 가입한 장치들에게만 전송
- IPv4: `224.0.0.0/4` 대역 사용
- 수신 원하는 장치가 IGMP로 라우터에 그룹 가입 신청
- 서버는 패킷을 1번만 전송 → 네트워크 장비가 가입 장치들에게 복사·전달
- ex) 실시간 스트리밍, IPTV, 화상회의, 주식 시세 전송
- 브로드캐스트는 관심 없는 장치도 강제로 받아야 함 vs 멀티캐스트는 가입한 장치만 받음

**애니캐스트** (Anycast)
- 동일 IP를 여러 서버가 공유 → 요청 시 가장 가까운 서버 하나에만 전달
- IPv6에서 주로 사용
- ex) CDN, DNS 루트 서버, DDoS 방어(공격 트래픽을 여러 서버로 분산)
- 한국 사용자는 자동으로 한국 서버, 미국 사용자는 미국 서버로 라우팅됨

| | 유니캐스트 | 브로드캐스트 | 멀티캐스트 | 애니캐스트 |
|---|---|---|---|---|
| 대상 | 1명 | 전체 | 그룹 | 최적 1명 |
| CPU 부하 | 낮음 | 높음 | 낮음 | 낮음 |
| 주 사용처 | 웹, 이메일 | ARP, DHCP | 스트리밍, IPTV | CDN, DNS |

## 8. TCP / UDP

| | TCP | UDP |
|---|---|---|
| 연결 방식 | 연결 지향 (3-way handshake) | 비연결 |
| 순서 보장 | O (시퀀스 번호) | X |
| 재전송 | O | X |
| 흐름·혼잡 제어 | O | X |
| 속도 | 느림 | 빠름 |
| 사용처 | HTTP, FTP, 이메일 | DNS, 스트리밍, 게임 |

- 왜 UDP를 쓰나? → 스트리밍·게임은 패킷 하나 유실됐을 때 재전송 기다리면 오히려 더 이상함
- 약간의 끊김은 허용하고 속도가 중요한 경우에 사용
- DNS는 요청/응답이 너무 짧아서 TCP 연결 과정이 오히려 더 느리므로 UDP 사용

## 9. OSI 7계층

네트워크 통신을 7개 계층으로 표준화한 모델
- 전송 시 상위 → 하위로 캡슐화
- 수신 시 하위 → 상위로 역캡슐화
- 계층별 역할 분리 → 한 계층에 문제가 생겨도 다른 계층에 영향 없음

| 계층 | 이름 | 데이터 단위 | 프로토콜·장치 | 역할 |
|---|---|---|---|---|
| 7 | 응용 | 데이터 | HTTP, FTP, DNS | 사용자 인터페이스 |
| 6 | 표현 | 데이터 | SSL/TLS, JPEG | 암호화, 압축, 형식 변환 |
| 5 | 세션 | 데이터 | NetBIOS, RPC | 세션 수립·유지·종료 |
| 4 | 전송 | 세그먼트 | TCP, UDP | 종단 간 신뢰 전송, 포트 번호 |
| 3 | 네트워크 | 패킷 | IP, ICMP | IP 기반 경로 결정 (라우팅) |
| 2 | 데이터링크 | 프레임 | 이더넷, 스위치 | MAC 기반 LAN 내 전송, 오류 검출 |
| 1 | 물리 | 비트 | 케이블, 허브 | 전기·광 신호로 비트 전송 |

- 암기법: **아파서 티나네 데물** (응용 - 표현 - 세션 - 전송 - 네트워크 - 데이터링크 - 물리)

## 10. 브리지 / 스위치

**브리지** (Bridge)
- OSI 2계층 장치. 두 LAN 세그먼트를 연결
- MAC 테이블로 필터링·포워딩
  - 목적지 MAC이 같은 세그먼트 → 차단 (필터링)
  - 목적지 MAC이 다른 세그먼트 → 전달 (포워딩)
- 허브는 무조건 모든 포트로 뿌림(1계층) vs 브리지는 MAC 테이블 보고 필요한 쪽만 전달(2계층)

**스위치** (Switch)
- 브리지의 발전형. 포트마다 전용 충돌 도메인
- MAC 주소 테이블(CAM Table) 유지
  - 목적지 MAC 알면 → 해당 포트로만 전달
  - 목적지 MAC 모르면 → 전체 포트로 플러딩 (Flooding)
- 하드웨어(ASIC) 처리 → 빠름

| | 브리지 | 스위치 |
|---|---|---|
| 포트 수 | 2~4개 | 수십~수백 개 |
| 처리 방식 | 소프트웨어 (느림) | 하드웨어 ASIC (빠름) |
| 현재 사용 | 거의 미사용 | LAN 표준 |

**L2 vs L3 스위치**
- L2 스위치: MAC 기반, 같은 네트워크 내 통신
- L3 스위치: IP 기반 라우팅도 가능, VLAN 간 라우팅 지원

- 스위치 vs 라우터: 스위치(2계층)는 같은 LAN 내 MAC 기반 전달 / 라우터(3계층)는 서로 다른 네트워크 간 IP 기반 경로 결정
