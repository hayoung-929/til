## Networking

### DHCP

DHCP(Dynamic Host Configuration Protocol)는 네트워크에 연결된 장치에 IP 주소, 서브넷 마스크, 게이트웨이, DNS 서버 주소를 자동으로 할당하는 프로토콜이다. UDP 67번(서버), 68번(클라이언트) 포트를 사용한다.

DHCP 할당 과정은 4단계로 이루어진다.

| 단계 | 메시지 | 방향 | 설명 |
|------|--------|------|------|
| 1 | DHCP Discover | 클라이언트 → 브로드캐스트 | IP 주소 없이 DHCP 서버 탐색 |
| 2 | DHCP Offer | 서버 → 클라이언트 | 사용 가능한 IP 주소 제안 |
| 3 | DHCP Request | 클라이언트 → 브로드캐스트 | 제안된 IP 주소 수락 요청 |
| 4 | DHCP ACK | 서버 → 클라이언트 | IP 주소 및 임대 정보 확정 |

IP 주소는 영구 할당이 아니며, 임대 시간(Lease Time)이 존재한다. 임대 시간이 만료되기 전에 갱신 요청을 보내고, 서버가 응답하지 않으면 IP를 반납한다.

### 서브넷 마스크

서브넷 마스크는 IP 주소에서 네트워크 부분과 호스트 부분을 구분하는 32비트 값이다. 연속된 1비트가 네트워크 영역, 이후의 0비트가 호스트 영역이다.

#### 클래스풀에서의 주소 구분

클래스풀(Classful) 방식은 IP 주소를 클래스 단위로 고정 분할한다.

| 클래스 | 첫 옥텟 범위 | 기본 서브넷 마스크 | 네트워크 비트 | 호스트 비트 |
|--------|-------------|-------------------|--------------|------------|
| A | 1 ~ 126 | 255.0.0.0 | 8 | 24 |
| B | 128 ~ 191 | 255.255.0.0 | 16 | 16 |
| C | 192 ~ 223 | 255.255.255.0 | 24 | 8 |

클래스풀 방식은 주소 낭비가 심해 현재는 CIDR로 대체되었다.

---

### CIDR 표기법

CIDR(Classless Inter-Domain Routing)는 서브넷 마스크를 `/프리픽스 길이` 형식으로 표기하는 방식이다. 클래스 경계에 구애받지 않고 임의의 비트 수로 네트워크를 분할할 수 있다.

```
192.168.1.0/24  →  서브넷 마스크: 255.255.255.0  (앞 24비트가 네트워크)
10.0.0.0/8      →  서브넷 마스크: 255.0.0.0       (앞 8비트가 네트워크)
172.16.0.0/12   →  서브넷 마스크: 255.240.0.0     (앞 12비트가 네트워크)
```

### 서브넷팅

서브넷팅(Subnetting)은 하나의 네트워크를 더 작은 여러 개의 서브네트워크(서브넷)로 분할하는 기법이다.

#### 서브넷팅의 목적

- **IP 주소 낭비 감소**: 필요한 호스트 수에 맞게 주소 공간을 할당한다.
- **브로드캐스트 도메인 분리**: 각 서브넷이 독립된 브로드캐스트 도메인을 형성하여 불필요한 트래픽을 줄인다.
- **보안 및 관리 분리**: 부서나 용도별로 네트워크를 논리적으로 구분할 수 있다.

#### 서브넷팅 계산법

호스트 비트에서 n비트를 빌리면 다음이 성립한다.

| 항목 | 공식 |
|------|------|
| 생성되는 서브넷 수 | 2ⁿ |
| 서브넷당 호스트 수 | 2^(남은 호스트 비트) − 2 |
| 네트워크 주소 | 호스트 비트가 전부 0 |
| 브로드캐스트 주소 | 호스트 비트가 전부 1 |
| 사용 가능한 호스트 범위 | 네트워크 주소 + 1 ~ 브로드캐스트 주소 − 1 |

−2를 하는 이유는 네트워크 주소와 브로드캐스트 주소는 호스트에 할당할 수 없기 때문이다.

### 문제

<details>
<summary>문제 1 — 192.168.1.0/24</summary>

**네트워크 주소**

프리픽스가 /24이므로 앞 24비트가 네트워크 영역이다. 호스트 비트(마지막 8비트)가 전부 0인 주소가 네트워크 주소다.

```
192.168.1.0
```

**브로드캐스트 주소**

호스트 비트가 전부 1인 주소다.

```
192.168.1.255
```

**사용 가능한 호스트 수**

호스트 비트가 8비트이므로:

```
2^8 − 2 = 256 − 2 = 254개
```

</details>

<details>
<summary>문제 2 — 192.168.1.0/26</summary>

**서브넷 마스크**

/26은 앞 26비트가 네트워크 영역이다. 마지막 옥텟 기준으로 26 − 24 = 2비트가 네트워크에 추가로 사용된다.

```
11111111.11111111.11111111.11000000
→ 255.255.255.192
```

**사용 가능한 호스트 수**

호스트 비트는 32 − 26 = 6비트:

```
2^6 − 2 = 64 − 2 = 62개
```

**첫 번째 호스트 ~ 마지막 호스트 범위**

```
네트워크 주소:    192.168.1.0
첫 번째 호스트:   192.168.1.1
마지막 호스트:    192.168.1.62
브로드캐스트:     192.168.1.63
```

</details>

<details>
<summary>문제 3 — 172.16.0.0/16을 4개의 서브넷으로 분할</summary>

**빌려야 하는 비트 수**

서브넷 수 ≥ 4가 되려면 2ⁿ ≥ 4이어야 하므로 n = 2비트를 빌린다.  
새 프리픽스: /16 + 2 = **/18**

**각 서브넷의 네트워크 주소**

빌린 2비트의 조합(00, 01, 10, 11)에 따라 서브넷이 결정된다.  
/18에서 세 번째 옥텟의 상위 2비트가 서브넷 식별자가 된다.

| 서브넷 | 네트워크 주소 | 브로드캐스트 주소 | 범위 |
|--------|--------------|-----------------|------|
| 1 (00) | 172.16.0.0/18 | 172.16.63.255 | 172.16.0.1 ~ 172.16.63.254 |
| 2 (01) | 172.16.64.0/18 | 172.16.127.255 | 172.16.64.1 ~ 172.16.127.254 |
| 3 (10) | 172.16.128.0/18 | 172.16.191.255 | 172.16.128.1 ~ 172.16.191.254 |
| 4 (11) | 172.16.192.0/18 | 172.16.255.255 | 172.16.192.1 ~ 172.16.255.254 |

세 번째 옥텟의 간격이 64씩 증가하는 이유는, /18에서 호스트 영역이 14비트(2^14 = 16384)이기 때문이다.

**각 서브넷당 호스트 수**

호스트 비트는 32 − 18 = 14비트:

```
2^14 − 2 = 16384 − 2 = 16382개
```

</details>
