## Frontend

- 사용자가 직접 보고 상호작용하는 화면을 만드는 역할
- 하는 일
  - 디자이너가 만든 Figma 시안을 실제 동작하는 웹/앱 화면으로 구현
  - 버튼 클릭, 폼 입력, 애니메이션 등 사용자 인터랙션 처리
  - 백엔드에서 받아온 데이터를 화면에 보기 좋게 렌더링
  - 반응형 웹 구현 (PC, 태블릿, 모바일 등 다양한 화면 크기에 맞게 레이아웃이 자동으로 조정되도록 구현)
  - 페이지 로딩 속도 최적화 (이미지 압축, 코드 분할, 캐싱 등을 활용해 사용자가 빠르게 화면을 볼 수 있도록 개선)
- 사용 기술
  - HTML : 웹 페이지의 구조를 만드는 언어
  - CSS : 색상, 폰트, 레이아웃 등 시각적인 스타일을 담당
  - JavaScript : 버튼 클릭, 데이터 처리 등 동적인 기능을 구현하는 언어
  - React / Vue / Angular : UI를 컴포넌트 단위로 관리하는 프레임워크. 컴포넌트란 버튼, 헤더처럼 재사용 가능한 UI 조각을 말함. 같은 버튼을 여러 페이지에서 쓸 때 한 번만 만들어두고 가져다 쓸 수 있어 유지보수가 쉬워짐
  - TypeScript : JavaScript에 타입을 추가해 코드 안정성과 가독성을 높인 언어

## Backend

- 사용자의 눈에는 보이지 않는 부분인 데이터 처리와 비즈니스 로직을 담당
- 하는 일
  - 프론트엔드의 요청을 받아서 데이터를 처리하고 응답
  - 데이터베이스에 데이터 저장, 조회, 수정, 삭제 (CRUD)
  - 회원가입, 로그인, 결제 같은 핵심 비즈니스 로직 구현
  - 보안 처리 : 비밀번호는 절대 그대로 저장하지 않고 해시(암호화)해서 저장. 로그인 후에는 JWT 토큰을 발급해서 로그인 상태를 유지함 (인증/인가)
  - API 설계 및 개발 : 프론트엔드가 데이터를 요청할 수 있는 창구(엔드포인트)를 만드는 것. 예를 들어 /api/login 이라는 주소를 만들어두면 프론트엔드가 그 주소로 요청을 보냄
- 사용 기술
  - Node.js / Express : JavaScript 기반 백엔드 프레임워크
  - Java / Spring : 대규모 기업 서비스에서 많이 사용하는 백엔드
  - Python / Django, FastAPI : 빠른 개발과 AI 연동에 강점이 있는 백엔드
  - MySQL / PostgreSQL : 데이터를 표(테이블) 형태로 저장하는 관계형 DB. 서로 관련된 데이터를 연결해서 체계적으로 관리할 수 있음
  - MongoDB : 데이터를 JSON 형태로 저장하는 비관계형 DB. 구조가 자유로워 빠른 개발에 유리함
  - Redis : 자주 조회되는 데이터를 메모리에 임시 저장해서 DB 부하를 줄이고 속도를 높이는 캐싱 도구

## 클라우드 / DevOps

- 서비스가 안정적으로 운영되도록 인프라를 구축하고 관리하는 역할
- 하는 일
  - 서버 구축 및 관리 : 직접 서버를 사는 대신 AWS, GCP 같은 클라우드 플랫폼을 활용해 서버를 구축하고 운영
  - 배포 자동화 (CI/CD) : 개발자가 GitHub에 코드를 올리면 자동으로 테스트를 실행하고, 문제가 없으면 서버에 자동 배포되는 파이프라인을 구축함. 사람이 직접 서버에 올리는 수동 작업을 없애 실수를 줄이고 배포 속도를 높임
  - 서버 모니터링 : 서비스가 다운되거나 오류가 발생하면 즉시 알림을 받고 빠르게 대응
  - 트래픽 급증 시 서버 자동 확장 (오토 스케일링) : 평소엔 서버 1대로 운영하다가 갑자기 사용자가 몰리면 자동으로 서버를 늘려 대응. 블랙프라이데이 같은 이벤트 때 서버가 터지지 않도록 방지
  - 보안 설정, 네트워크 구성, 비용 최적화
- 사용 기술
  - AWS / GCP / Azure : 서버, 스토리지, 네트워크 등을 제공하는 클라우드 플랫폼
  - Docker : 애플리케이션과 실행 환경을 하나의 컨테이너로 묶어서 패키징. 환경 차이 문제를 해결해줌
  - Kubernetes : Docker 컨테이너가 여러 개일 때 자동으로 배포, 복구, 확장을 관리해주는 도구
  - CI/CD (GitHub Actions, Jenkins) : 코드 변경 시 자동으로 테스트하고 배포하는 자동화 파이프라인
  - Terraform : 서버, 네트워크 등 인프라 설정을 코드로 작성해서 관리. 클릭으로 설정하던 걸 코드로 자동화함
  - Linux : 대부분의 서버에서 사용하는 운영체제

## Design

- 사용자가 서비스를 편하고 직관적으로 사용할 수 있도록 화면을 설계하는 역할
- 하는 일
  - 사용자 리서치 : 어떤 기능이 필요한지, 사용자가 어떤 방식으로 서비스를 쓰는지 조사하고 분석
  - 와이어프레임 제작 : 색상이나 디자인 없이 화면의 레이아웃과 구조만 잡아놓은 초안 스케치
  - UI 디자인 : 실제 색상, 폰트, 아이콘 등 시각적으로 보이는 요소들을 설계
  - UX 설계 : 사용자가 원하는 목표(예: 상품 구매)까지 도달하는 흐름을 설계. 불필요한 단계를 줄이고 직관적으로 사용할 수 있도록 고민하는 것이 핵심
  - 프로토타입 제작 : 실제로 클릭하고 이동할 수 있는 데모 화면을 만들어 개발 전에 미리 검토
  - 디자인 시스템 구축 : 버튼 색상, 폰트 크기, 간격 등을 규칙으로 정해두는 것. 팀 전체가 같은 규칙을 따르면 디자인이 일관성 있게 유지되고, 개발자도 기준을 보고 빠르게 구현할 수 있음
- 사용 도구
  - Figma : 현재 가장 많이 사용되는 UI/UX 디자인 협업 툴
  - Adobe XD / Illustrator / Photoshop : 그래픽 작업 및 상세 디자인에 활용
  - Zeplin : 디자이너가 만든 시안을 개발자에게 전달할 때 색상 코드, 폰트 크기, 간격 등의 정보를 자동으로 제공하는 협업 툴

<summary>프론트엔드 ↔ 백엔드 협업 방식</summary>

## 프론트엔드 ↔ 백엔드 협업 방식

프론트엔드와 백엔드는 서로 역할이 분리되어 있기 때문에, 둘 사이에 데이터를 주고받는 방식을 미리 약속해두어야 한다. 이 약속을 **API**라고 하며, 협업은 이 API를 중심으로 이루어짐

### 1. API 명세서 작성

백엔드가 먼저 API를 설계하고 명세서를 작성한다. API 명세서란 "어떤 주소로 요청을 보내면, 어떤 데이터를 돌려준다"는 내용을 정리한 문서이다. 예를 들어 `/api/login` 주소로 아이디와 비밀번호를 보내면 로그인 성공 여부를 응답한다는 식으로 작성한다. 프론트엔드는 이 명세서를 보고 화면을 개발하기 때문에, 명세서가 먼저 작성되어야 협업이 원활하게 진행됨

### 2. 프론트엔드의 HTTP 요청

프론트엔드는 명세서를 보고 사용자의 행동(버튼 클릭, 폼 제출 등)이 발생했을 때 백엔드로 HTTP 요청을 보내는 코드를 작성한다. 예를 들어 사용자가 로그인 버튼을 누르면 `/api/login` 주소로 아이디와 비밀번호를 담아 POST 요청을 보내는 식이다.

### 3. 백엔드의 데이터 처리 및 응답

백엔드는 프론트엔드로부터 요청을 받으면 데이터베이스에서 필요한 데이터를 조회하거나 저장한 뒤, 결과를 JSON 형태로 응답한다. JSON은 데이터를 key-value 쌍으로 표현하는 형식으로, 프론트엔드와 백엔드가 데이터를 주고받을 때 가장 많이 사용됨

### 4. 프론트엔드의 데이터 렌더링

프론트엔드는 백엔드로부터 받은 JSON 데이터를 가공해서 화면에 렌더링한다. 예를 들어 사용자 목록 데이터를 받으면 그것을 표나 카드 형태로 화면에 출력하는 식이다.

### 5. Mock 데이터를 활용한 병렬 개발

실제 개발 과정에서는 백엔드 API가 완성되기 전에 프론트엔드 개발을 먼저 시작해야 하는 경우가 많다. 이때 Mock 데이터(실제 API 대신 사용하는 가짜 데이터)를 활용하면 프론트엔드와 백엔드가 서로 기다리지 않고 동시에 개발을 진행할 수 있다. API가 완성되면 Mock 데이터를 실제 API 호출로 교체하기만 하면 됨됨

### 6. 배포

개발이 완료되면 클라우드/DevOps가 프론트엔드와 백엔드를 서버에 배포하여 실제 사용자가 접근할 수 있는 상태로 만든다.

# HTTP란?
<details>
<summary>HTTP란?</summary>

## HTTP란?

HTTP(HyperText Transfer Protocol)는 웹에서 클라이언트와 서버가 데이터를 주고받을 때 사용하는 통신 규약이다. 여기서 통신 규약이란 데이터를 주고받는 방식을 미리 약속해둔 규칙이라고 생각하면 된다. 우리가 브라우저 주소창에 URL을 입력하고 엔터를 누르는 순간, 브라우저는 해당 서버에 HTTP 요청을 보내고 서버는 HTTP 응답으로 웹 페이지 데이터를 돌려준다. 웹 페이지, 이미지, JSON 데이터 등 대부분의 웹 리소스가 이 HTTP를 통해 전송된다.

</details>

<details>
<summary>Stateless 구조</summary>

## Stateless 구조

HTTP는 Stateless(무상태) 프로토콜이다. Stateless란 서버가 클라이언트의 이전 요청 상태를 기억하지 않는다는 의미이다. 매 요청은 완전히 독립적으로 처리되며, 서버 입장에서는 요청이 올 때마다 처음 보는 요청으로 인식한다.

예를 들어 로그인을 한 뒤 다음 페이지로 이동하면서 새로운 요청을 보내도, 서버는 이전에 로그인했다는 사실을 기억하지 못한다. 이 문제를 해결하기 위해 아래와 같은 방법으로 상태 정보를 별도로 관리한다.

- **쿠키** : 브라우저에 데이터를 저장해두고, 매 요청마다 자동으로 서버에 함께 전송하는 방식
- **세션** : 서버에 로그인 상태를 저장해두고, 클라이언트에는 세션 ID만 발급하는 방식
- **JWT 토큰** : 로그인 성공 시 서버가 토큰을 발급하고, 클라이언트가 이후 요청마다 토큰을 함께 보내는 방식

Stateless 구조 덕분에 서버는 각 요청을 독립적으로 처리할 수 있어 확장성이 높아진다. 서버가 여러 대로 늘어나도 특정 서버에 종속되지 않고 어느 서버에서나 요청을 처리할 수 있기 때문이다.

</details>

<details>
<summary>클라이언트-서버 구조</summary>

## 클라이언트-서버 구조

HTTP는 클라이언트-서버 구조를 기반으로 동작한다. 클라이언트(브라우저, 앱)가 먼저 요청(Request)을 보내고, 서버가 그에 대한 응답(Response)을 돌려주는 방식이다. 서버는 요청이 오기 전까지 먼저 데이터를 보내지 않으며, 항상 클라이언트의 요청이 있어야만 응답한다.

이 구조의 핵심은 역할이 명확하게 분리되어 있다는 점이다. 클라이언트는 화면을 보여주는 것에 집중하고, 서버는 데이터를 처리하는 것에 집중한다. 덕분에 클라이언트와 서버를 독립적으로 개발하고 배포할 수 있으며, 서버 하나에 웹 브라우저, 모바일 앱 등 다양한 클라이언트를 연결하는 것도 가능하다.

</details>

<details>
<summary>HTTP 메서드</summary>

## HTTP 메서드

HTTP 메서드는 클라이언트가 서버에 어떤 작업을 요청하는지 나타내는 방식이다. 같은 주소라도 어떤 메서드를 사용하느냐에 따라 서버가 수행하는 동작이 달라진다. 예를 들어 `/api/post`라는 주소에 GET 요청을 보내면 게시글 목록을 조회하고, POST 요청을 보내면 새 게시글을 작성하는 식이다.

| 메서드 | 설명 |
|--------|------|
| `GET` | 리소스 조회. 서버의 상태를 변경하지 않음 |
| `POST` | 새로운 리소스 생성 또는 서버로 데이터 전송 (회원가입, 글쓰기 등) |
| `PUT` | 리소스를 통째로 교체. 리소스가 없으면 새로 생성 |
| `PATCH` | 리소스의 일부만 수정 |
| `DELETE` | 리소스 삭제 |

</details>

<details>
<summary>HTTP 상태 코드</summary>

## HTTP 상태 코드

HTTP 상태 코드는 서버가 클라이언트의 요청을 처리한 결과를 숫자로 나타낸 것이다. 클라이언트는 이 숫자를 보고 요청이 성공했는지, 실패했는지, 실패했다면 어떤 이유인지 파악할 수 있다. 상태 코드는 첫 번째 숫자에 따라 범위별로 의미가 구분된다.

| 범위 | 의미 |
|------|------|
| 2xx | 요청 성공 |
| 3xx | 리다이렉션 (다른 URL로 이동 필요) |
| 4xx | 클라이언트 오류 (잘못된 요청) |
| 5xx | 서버 오류 (서버 내부 문제) |

4xx는 클라이언트가 잘못된 요청을 보낸 경우이고, 5xx는 요청 자체는 정상이지만 서버에서 처리하다가 문제가 생긴 경우이다. 즉, 4xx는 클라이언트 잘못, 5xx는 서버 잘못이라고 구분하면 이해하기 쉽다.

| 코드 | 설명 |
|------|------|
| `200 OK` | 요청이 정상적으로 처리됨 |
| `201 Created` | 요청이 성공하여 새로운 리소스가 생성됨 |
| `301 Moved Permanently` | 요청한 리소스가 영구적으로 다른 URL로 이동됨 |
| `400 Bad Request` | 클라이언트가 잘못된 형식으로 요청을 보냄 |
| `401 Unauthorized` | 인증이 필요하거나 인증에 실패함 |
| `403 Forbidden` | 인증은 됐지만 해당 리소스에 접근 권한이 없음 |
| `404 Not Found` | 요청한 리소스가 서버에 존재하지 않음 |
| `500 Internal Server Error` | 서버 내부 오류로 요청을 처리하지 못함 |

</details>

## HTTP 요청/응답 구조

HTTP 메시지는 Header와 Body로 구성된다. 요청(Request)과 응답(Response) 모두 이 구조를 따른다.

### Header

Header는 요청 또는 응답에 대한 메타데이터를 담고 있다. 메타데이터란 실제 데이터는 아니지만, 데이터를 올바르게 처리하기 위해 필요한 부가 정보를 말한다. 예를 들어 편지를 보낼 때 편지 내용이 Body라면, 봉투에 적힌 수신자 주소, 발신자 정보, 우표 같은 것이 Header에 해당한다.

Header에는 아래와 같은 정보들이 포함된다.

- **Content-Type** : 전송되는 데이터의 형식을 나타냄. `application/json`이면 JSON 형식, `text/html`이면 HTML 형식임을 서버에 알려줌
- **Authorization** : 인증 토큰 정보. 로그인한 사용자임을 서버에 증명할 때 사용
- **Content-Length** : Body 데이터의 크기
- **Cache-Control** : 응답 데이터를 브라우저가 캐시(임시 저장)할지 여부를 제어

### Body

Body는 실제로 전송되는 데이터를 담고 있다. 모든 요청에 Body가 있는 것은 아니다. `GET` 요청처럼 단순히 데이터를 조회하는 경우에는 Body가 없으며, `POST`나 `PUT` 요청처럼 서버에 데이터를 전송해야 하는 경우에 Body에 데이터를 담아 보낸다. 응답의 경우 서버가 돌려주는 JSON 데이터, HTML 페이지, 이미지 파일 등이 Body에 담긴다.
